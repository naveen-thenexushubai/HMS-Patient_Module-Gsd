# Hospital Management System - Patient Module

## What This Is

A patient management system that enables hospital staff (receptionists, doctors, nurses, administrators) to register, search, view, and manage patient demographic information and medical history. This module serves as the foundation for a larger hospital management system with 40+ modules, providing centralized patient data that other modules (appointments, EMR, billing) will depend on.

## Core Value

Centralized, secure, and efficient patient information management that serves as the foundation for all other hospital modules. If patient data isn't accurate and accessible, nothing else in the hospital system works.

### Active

- [ ] Patient Registration - Receptionists can register new patients with complete demographic information (name, DOB, gender, phone, email, address, emergency contacts, medical info)
- [ ] Patient Search and Filtering - Staff can search patients by multiple criteria (ID, name, phone, email) and filter by status, gender, blood group with real-time results
- [ ] Patient Profile View - Staff can view complete patient information in organized layout including demographics, emergency contacts, and medical background
- [ ] Patient Information Update - Receptionists and admins can update patient demographic information with full validation
- [ ] Patient Status Management - Admins can activate/deactivate patient records instead of deleting them to maintain historical data
- [ ] Patient ID Generation - System generates unique patient IDs in format "P" + year + sequential number (e.g., P2026001)
- [ ] Role-based Access Control - Different permissions for Receptionist (full CRUD), Doctor/Nurse (read-only), Admin (full access including deactivation)
- [ ] Audit Trail - System records who registered/updated each patient and when
- [ ] Duplicate Detection - System warns about potential duplicates when registering patients with existing phone numbers
- [ ] Data Validation - Phone number format validation, email format validation, date of birth validation, auto-calculate age

### Out of Scope

- Authentication and user login (Auth Module) — separate module will handle user authentication and role management
- Appointment scheduling (Appointment Module) — patient appointments handled separately
- Medical records and clinical notes (EMR Module) — clinical documentation separate from demographics
- Prescriptions and medications (EMR/Pharmacy Module) — medication management out of scope
- Lab results and imaging (Laboratory/Radiology Module) — diagnostic data separate
- Billing and insurance (Billing Module) — financial operations separate
- Patient portal for self-service (Patient Portal Module) — patient-facing features separate
- Document upload and management (Document Module) — file management separate
- Notifications and reminders (Notification Module) — communication features separate
- Reporting and analytics (Reporting Module) — business intelligence separate

## Context

**Exploration Context:**
- Part of existing 40+ module enterprise hospital management system
- Exploring GSD workflow to potentially rebuild entire system from scratch
- Starting with Patient Module as foundation since other modules depend on patient data
- Current requirements generated by AI tool and need validation through actual implementation

**Healthcare Context:**
- Patient data is Protected Health Information (PHI) under HIPAA
- All access must be logged for audit compliance
- Data accuracy is critical for patient safety
- System will be used by multiple staff types with different needs (fast registration for receptionists vs quick lookup for doctors)

**Technical Context:**
- Microservices architecture expected
- RESTful APIs for module integration
- Separate frontend and backend
- Database-first design with strong data integrity

**Scale Requirements:**
- Support up to 50,000 patient records
- Handle up to 100 concurrent users
- Patient search must return results within 2 seconds
- Registration must complete within 3 seconds

## Constraints

- **Tech Stack**: Spring Boot 3.4.5+ with Java 17 (backend), React 18.x (frontend), PostgreSQL 15+ (database) — chosen for enterprise requirements (upgraded from 3.2.x due to CVE-2025-22235)
- **Performance**: Search results within 2 seconds for 10,000 records, registration within 3 seconds — user experience requirement
- **Security**: HIPAA compliance mandatory, all patient data is PHI — legal requirement
- **Scalability**: Must support 50,000 patients and 100 concurrent users — operational requirement
- **Data Integrity**: Patient ID must be unique and immutable, no deletion of patient records (soft delete only) — audit trail requirement
- **Validation**: Phone numbers, emails, dates must be validated before storage — data quality requirement
- **Audit**: All access and modifications must be logged with user and timestamp — compliance requirement
