Implement JPA AttributeConverter for transparent field-level encryption of SSN and insurance policy numbers.

**EncryptionConfig.java:**
```java
@Configuration
public class EncryptionConfig {
    @Value("${encryption.key}")
    private String encryptionKeyBase64;

    @Bean
    public SecretKey encryptionKey() {
        byte[] decodedKey = Base64.getDecoder().decode(encryptionKeyBase64);
        if (decodedKey.length != 32) {
            throw new IllegalStateException("Encryption key must be 32 bytes (256 bits)");
        }
        return new SecretKeySpec(decodedKey, 0, decodedKey.length, "AES");
    }
}
```

**EncryptionService.java:**
- @Service class
- Inject SecretKey bean from EncryptionConfig
- encrypt(String plaintext): String
  - Use AES/GCM/NoPadding (authenticated encryption)
  - Generate random 12-byte IV for each encryption
  - Prepend IV to ciphertext: Base64.encode(IV + ciphertext)
  - Return base64-encoded result
- decrypt(String ciphertext): String
  - Base64 decode input
  - Extract first 12 bytes as IV
  - Decrypt remaining bytes
  - Return plaintext
- Handle null inputs (return null)
- Throw IllegalStateException on encryption/decryption failure

**SensitiveDataConverter.java:**
- @Converter(autoApply = false)  // NOT auto-apply - explicit usage on fields
- Implements AttributeConverter<String, String>
- Inject EncryptionService
- convertToDatabaseColumn(String attribute): Calls encryptionService.encrypt()
- convertToEntityAttribute(String dbData): Calls encryptionService.decrypt()
- Handle null values properly

Use patterns from 00-RESEARCH.md Pattern 1. Use AES/GCM/NoPadding not AES/CBC (GCM provides authentication, prevents tampering). Store IV with ciphertext (no need for separate IV column).

Do NOT create entity classes yet (Phase 1). This converter will be used with @Convert(converter = SensitiveDataConverter.class) on SSN and insurance policy fields.
